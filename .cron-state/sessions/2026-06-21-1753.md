## Field summary

### New ReceiptFields (4)

- `service_charge: float | None` -- explicit "Service Charge"/"Service
  Fee"/"Svc Charge"/"Svc Fee" amount. Distinct from `tip`. Bare
  "Service" stays in `_TIP_KEYWORDS` for backward-compat with UK
  bar-tab semantics.
- `delivery_fee: float | None` -- "Delivery Fee"/"Delivery Charge"/
  "Delivery"/"Shipping"/"Shipping Fee"/"Shipping & Handling"/"Shipping
  and Handling"/"Shipping Charge"/"Shipping Cost". Multi-word forms
  beat bare aliases. Recognised on UberEats/DoorDash/Deliveroo,
  Amazon/Shopify, Instacart.
- `tendered: float | None` -- cash handed to cashier. Catalogue (most-
  specific first): "Cash Tendered"/"Tendered"/"Tender"/"Amount
  Tendered"/"Amount Paid"/"Paid"/"Payment"/"Cash".
- `change: float | None` -- change handed back. "Change Due"/"Change
  Given"/"Cash Change"/"Change". "Change 0.00" intentionally
  registers (meaningful for till-discrepancy dashboards).

### New ReceiptLine field (1)

- `sku: str | None` -- stock-keeping unit / barcode / UPC / EAN /
  item-code / PLU printed alongside the line item. Inline form
  "Latte SKU: 12345 5.00" (cleaned to "Latte 5.00" with sku=12345
  attached). Standalone form "SKU: 12345" on its own line attaches
  to the most-recent item. Recognised wording: SKU/Barcode/UPC/EAN/
  GTIN/PLU/Item Code/Item No./Item #/Item Number. Value charset:
  alphanumerics + dashes/underscores/dots/slashes, 3..32 chars.

### New cross-category raw keys (2)

- `raw["slack_ids"]` -- list of `{kind, id}` dicts. 10 Slack ID
  prefixes mapped to long-form kind tags (C->channel, D->dm,
  G->private_channel, U->user, W->enterprise_user, B->bot, T->team,
  E->enterprise, F->file, S->usergroup). Tail must contain at least
  one digit so all-letter prose ("CHEAPCODE", "DESPAIRED") doesn't
  misfire. Word-boundary isolation on both ends. Slack mention
  syntax `<@U..>` / `<#C..|name>` / `<!subteam^S..>` handled
  naturally.
- `raw["crypto"]` -- list of `{chain, address}` dicts. Three chains:
  - bitcoin: Base58Check P2PKH/P2SH validated with 4-byte
    SHA256(SHA256(payload))[:4] checksum, AND Bech32/Bech32m
    SegWit/Taproot validated with BCH polymod against the right
    constant per witness version (1 for v0, 0x2bc830a3 for v1+).
  - ethereum: 0x + 40 hex shape-only (EIP-55 needs keccak256
    outside stdlib). All-zero null address rejected. Lowercased.
  - solana: 32..44 Base58 shape-only AND requires Solana-context
    anchor (sol/solana/spl/phantom/mint/pubkey/wallet/token) on
    same or previous line because Base58 alphabet overlaps with
    random base58-shaped IDs.

  Pure-Python implementation, no new heavy deps.

## Notable design decisions

- service_charge intentionally NOT routed through the legacy `tip`
  field so a restaurant receipt's mandatory "Service Charge 5.00"
  PLUS voluntary "Tip 4.00" populates BOTH fields. The bare "Service"
  alias stays in `_TIP_KEYWORDS` for UK bar-tab compat -- breaking
  that would break the existing `test_tip_extracted_from_service_charge`
  test that's been green since tick 1.
- Tendered/change "Cash" matcher would naively fire on "Cashier #04"
  but doesn't because the underlying `_find_amount_after` regex
  requires a digit-amount IMMEDIATELY after the keyword (with at most
  `:` / `-` separators), and "Cashier #04" has the "ier #" between
  "Cash" and "04". Test pinned for that case.
- SKU extractor strips the keyword + value off the line BEFORE the
  per-item parsers fire so a "Latte SKU: 12345 5.00" line parses
  cleanly as "Latte 5.00" with sku=12345 attached. Standalone
  "SKU: 12345" lines (no price tail) attach to the most-recent item.
  All four per-item parser branches (pct-off, qty-prefixed, @-form,
  bare desc+price) attach the extracted SKU.
- Slack ID extractor's "tail must contain at least one digit" rule
  is the discriminator that keeps all-letter prose words like
  "CHEAPCODE" and "DESPAIRED" from misfiring as channel IDs. Real
  Slack IDs are random base32-ish strings that empirically always
  carry a digit somewhere.
- Crypto extractor's BTC base58check scanner runs FIRST so a 34-char
  address that satisfies the Solana shape too is tagged bitcoin (not
  double-tagged). Solana's anchor requirement is documented as the
  cost of recall -- without the anchor, base58 alphabet overlap
  with random IDs makes false-positives unbounded.
- Pure-Python crypto helpers (Base58 decode + double-SHA256 + bech32
  polymod) -- no new heavy deps. Validated against canonical test
  vectors (Genesis P2PKH, BIP-173 SegWit, BIP-350 Taproot).
