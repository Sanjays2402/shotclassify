## What landed

### 1. JWT extractor (raw["jwts"])
New cross-category extractor surfacing JSON Web Tokens. Decodes the JOSE header + standard registered payload claims (iss/sub/aud/exp/iat/nbf/jti). **Security guarantee**: the FULL token is NEVER stored — signature segment discarded entirely; custom payload claims (email/preferred_username/custom_attr) intentionally NOT surfaced because tokens carry PII inside custom claims. Pairs with the existing `jwt` redact mode for defence-in-depth. String claims capped at 256 chars; list aud collapses to comma-joined string; float exp/iat that are whole numbers collapse to int. Cap 20 entries.

### 2. Markdown fence-language detection (CodeFields.fence_language)
Surfaces the lowercased language tag from a markdown code fence (```python / ~~~rust / 4+ backticks). CommonMark + GFM compatible: handles up-to-3-space indentation rules, info-string suffixes (title="..." / hl_lines=...), bare-fence rejection. First tag wins when multiple fences differ. We do NOT canonicalise short forms (js stays js, py stays py) because the original tag carries author intent. Runs FIRST in enrich_code on the pre-strip body so fence markers survive line-number stripping.

### 3. Feature-flag SDK detection (CodeFields.feature_flags)
8 vendors catalogued with their canonical SDK shapes: LaunchDarkly, Statsig, Unleash, Optimizely, Split.io, PostHog, Flagsmith, ConfigCat. Each entry is a `{vendor, key}` dict. Patterns are vendor-specific so each call site matches exactly one vendor (no ambiguity). Flag-key character set is `[A-Za-z][A-Za-z0-9._-]{0,127}` so dashed/dotted/snake_case all parse. Distinct from `imports` (library dependency) — this slot tracks per-call flag-key references.

### 4. CSS vendor-prefix detection (CodeFields.css_vendor_prefixes)
5 recognised prefixes: -webkit-, -moz-, -ms-, -o-, -khtml-. Entries include leading and trailing hyphen so the output is directly usable. Language-gated to CSS-family (css/scss/sass/less/stylus) with a **content fallback** that fires when the snippet contains BOTH a vendor-prefix candidate AND a CSS-like property declaration within 200 chars — catches pygments mis-classified CSS bodies (it sometimes returns `gas` / `text` for short CSS) without false-positiving on JS comments that mention `-webkit-`.

### 5. Passport-number redaction mode (`passport`)
Strips labelled passport numbers. Requires the word `passport` immediately before the candidate so bare 9-digit runs don't misfire on order numbers / phones. Recognises US/UK 9-digit + Australia 1-letter+7-digit + Germany 1-letter+8-digit + Canada 2-letter+6/7-digit + mixed alphanumeric shapes. Custom `_sub_passport` substitution preserves the `Passport: ` label and replaces only the captured `num` span (reader knows the field WAS a passport without the number leaking). 11+ digit runs fail the trailing word-boundary and are LEFT UNCHANGED as a safety property. Added to PII_REDACT_MODES allow-list.

## Notable design decisions

- **JWT signature discarded as security property**: explicitly tested. We could decode and surface the signature byte length, but even that is too risky if the consumer logs it. Best to drop it entirely.
- **Markdown fence detector runs first**: before line-number stripping so fence markers (``` / ~~~) survive any downstream code transformation.
- **Feature-flag vendor disambiguation via prefix**: every SDK uses a distinctive prefix (`ldClient.` / `posthog.` / `optimizely.` / etc.) so the per-vendor regexes don't overlap. A single call site can only match one vendor.
- **CSS content fallback is LOCAL not global**: 200-char window per candidate. A faraway random `color: red;` declaration in a JS file doesn't trigger; only candidates with a NEARBY declaration qualify.
- **Passport label preservation**: chose to preserve the `Passport: ` label instead of replacing the entire match. The reader gains semantic context without leaking the number itself.
