### 1. Error: Vue.js component error parsing (`framework='vue'`)
Commit: `5cf837c`

Detection requires literal `[Vue warn]:` prefix combined with a
recognised Vue slot. 8 recognised slot families:
1. `Error in v-on handler: "..."` (event handlers)
2. `Error in render` / `Error in render function` (template rendering)
3. `Error in <lifecycle> hook: "..."` (15 hooks: beforeCreate, created,
   beforeMount, mounted, beforeUpdate, updated, activated, deactivated,
   beforeUnmount, unmounted, beforeDestroy, destroyed, errorCaptured,
   renderTracked, renderTriggered, serverPrefetch)
4. `Error in callback for watcher "..."` (Vue 2/3 watchers)
5. `Error in setup function` (Vue 3 composition API)
6. `Error in directive <name> hook` (custom directives)
7. `Unhandled error during execution of <slot>` (Vue 3 default handler)
8. `Hydration <kind> mismatch` (Vue 3 SSR/SSG -- node/text/class/style/
   attribute/children)

Branch placement: BEFORE Node branch because Vue runs on JS runtime and
the bare `_JS_AT` pattern would otherwise steal a Vue capture's JS
stack tail.

Exception slot resolution priority:
- Quoted inner exception wins when present: `"TypeError: x is not
  defined"` -> exc="TypeError"
- Hydration mismatch becomes `HydrationNodeMismatch` / `HydrationText
  Mismatch` etc.
- Unhandled error wraps slot in `VueUnhandledError(<slot>)`
- Otherwise the slot becomes a `Vue<Slot>Error` class name

Quoted-error regex was tightened mid-implementation to make the
exception-class prefix OPTIONAL so bare `"Error: msg"` lands as
exc="Error". Caught by test_vue_created_hook failing on first run.

File slot: innermost `.vue` file from the `found in` component tree
(`---> <Foo> at src/Foo.vue`) with fallback to `<ComponentTag>` for
either arrow-prefixed entries OR bare `at <Tag>` entries (the latter
shape comes from Vue 3's Unhandled error handler).

Line slot: always None because Vue's warn handler doesn't print
per-frame line numbers.

20-cause `_vue_likely_cause` catalogue with TYPED-exception hints
checked BEFORE slot hints so TypeError+undefined surfaces "optional
chaining" hint instead of generic "mounted hook" hint -- the typed
hint is more actionable for triage.

Public alias `parse_vue_error()` exposed (mirrors `parse_swift_crash` /
`parse_apollo_error` / `parse_kotlin_coroutine` convention) and added
to `__all__`.

Tests: 38 covering all recognised slot shapes, file/component
attribution, Vue 2 + Vue 3 variations, hydration mismatch kinds,
likely-cause hint dispatch, branch ordering (Vue wins over Node, plain
Node error rejects), case-insensitive prelude, empty/None input.

### 2. Document: heading-hierarchy detection (`DocumentFields.headings`)
Commit: `da83251`

A new `DocumentFields.headings` slot captures the H1..H6 outline of
multi-page document captures (slide decks, scanned reports, wiki
pages, technical specs, contracts). Output is a list of
`{"level": int, "text": str}` dicts where level is 1..6.

Three recognised shape families, scanned in priority order per-line
with a claim-set so the same line doesn't double-count:

1. **Markdown ATX**: `# Heading` to `###### Heading`. The hash run
   must be 1..6 (7+ rejected -- CommonMark spec). Optional trailing
   closing-hash run stripped.

2. **Markdown setext**: text line followed by `===` (3+ chars) -> h1
   OR `---` (3+ chars) -> h2. Backreference `\1{2,}` enforces
   same-character runs so `=-=-=` rejects.

3. **Numbered** (contract/spec convention): `1. Chapter` -> h1,
   `1.1 Section` -> h2, `1.1.1 Subsection` -> h3, depth capped at 6.
   Body must start with letter or opening quote (so pure-numeric
   quantities reject), body >80 chars rejected (long prose),
   body starting with `#` rejects (avoids ATX-inside-numbered
   double-counting).

Two regex bugs caught + fixed on first gate run:
- Setext divider `(=|-){3,}` matched `=-=-=` because alternation
  was per-char; switched to backreference `\1{2,}` enforcing
  same-character runs.
- Numbered regex `{0,5}` segments was too tight for the
  `1.1.1.1.1.1.1` (7-segment) test; expanded to `{0,9}` and
  let `_level_for_numbered` cap at 6 downstream.

Output sorted by source-text appearance offset. Cap 100 entries.

`enrich_document` plumbing: caller's non-empty headings preserved
(LLM may surface richer structure), empty backfilled from regex.
LLM wire format uses `DocumentFields.model_fields` kwargs splat so
`headings` passes through automatically.

Tests: 51 covering all three shape families, mixed-shape documents,
source-text ordering preservation, the per-line claim set, safety
rejections, 100-cap behaviour, and 4 real-world fixtures (slide
deck, legal contract, README, wiki page).

### 3. Extract: cross-category color extractor (`raw["colors"]`)
Commit: `4a63c82`

New `ExtractedFields.raw["colors"]` slot stashes every recognised
colour value as a list of `{"model": str, "value": str}` dicts.

9 recognised colour models:
- `hex`: `#FF5733` / `#fa3` short / `#FF5733AA` alpha / `0xFF5733`
- `rgb`: comma OR space (CSS-4) OR slash-alpha forms
- `hsl`: with deg/rad/turn/grad units on hue, negative hue accepted
- `hsv`: non-CSS but common in design tools
- `oklch` / `oklab` / `lab` / `lch`: CSS-4 perceptual spaces
- `named`: ~100 curated CSS named colours

**Safety property**: common dictionary words (red / blue / green /
black / white / yellow / grey / gray) are INTENTIONALLY EXCLUDED
from the named catalogue because they appear far too often in prose
to be safe matchers. Only visually-AND-lexically distinctive names
(rebeccapurple / coral / mediumaquamarine etc) land in the catalogue.

Hex matcher requires `#`/`0x` prefix combined with word-boundary so
bare `FF5733` in OAuth state / session ID / UUID rejects. Function-
form matchers require function name + `(` so prose mentions of the
bare word "rgb" don't fire. RGB channel validation 0..255 with
`rgb(256, ...)` rejecting.

Named catalogue uses longest-first ordering so `mediumaquamarine`
beats `aquamarine`. Span-claim defence prevents hex inside function
calls from double-counting.

Dedup on (model, value) pair preserving first-seen-order. Cap 100.

Tests: 61 covering every recognised model, the curated-vs-prose
distinction, CSS rule blocks / Tailwind config / design-system doc /
Figma inspector fixtures, dedup behaviour, source-text ordering,
100-cap behaviour, pipeline integration across categories, and
safety rejections.

### 5. Extract: cross-category emoji-density (`raw["emoji_density"]`)
Commit: `f52de90`

New `ExtractedFields.raw["emoji_density"]` slot captures a single
float in `[0.0, 1.0]` representing the share of non-whitespace
characters in the OCR text that participate in an emoji codepoint
sequence. A quick "this capture is meme-heavy" signal.

Numerator counts every base emoji codepoint PLUS modifier glue
(skin-tone, variation selector, ZWJ + next-base) so a ZWJ family
emoji like the family-of-four contributes 7 codepoints (4 base + 3
ZWJs) to the numerator instead of just 1.

Denominator excludes whitespace because OCR captures vary wildly
in whitespace preservation; dashboards comparing captures should
be comparing meaningful content density, not padding density.

Return value semantics:
- `None` when text is empty / not a string (no signal)
- `0.0` when text has only non-emoji content (legitimate signal --
  distinct from None which means "couldn't compute")
- Float >0.0 when at least one emoji codepoint appears
- Clipped to `[0.0, 1.0]` defensively
- Rounded to 3 decimal places for stable storage

Pipeline writes the density even when `raw["emojis"]` is empty
because density=0.0 lets dashboards filter "emoji-free vs
emoji-heavy" captures consistently. Only empty / null OCR text
leaves the slot unset.

Reuses `_is_base_emoji` / `_is_skin_tone` / `_VARIATION_SELECTORS`
/ `_ZWJ` from `extract_emojis` so the codepoint-detection logic
stays in one place.

Distinct from `raw["emojis"]` (per-emoji tally) -- this is the
WHOLE-DOCUMENT aggregate score. Distinct from `chat.reactions`
(per-message footers) -- this runs cross-category.

Tests: 31 covering plain-text ratios, whitespace exclusion semantics,
compound emoji counting (ZWJ families, skin tones, variation
selectors), None/empty input guards, rounding behaviour, range
clipping, all 10 detected Unicode ranges firing positively, plain
ASCII symbols correctly NOT counted, and pipeline integration
across categories.

## What needed tightening

- The setext-divider regex bug was subtle: `(=|-){3,}` looks
  correct but the alternation matches each char independently
  so `=-=-=` qualified as 5 divider chars. The backreference
  `\1{2,}` fix is the right pattern for "3+ of the same char".
- The Vue likely-cause hint ordering needed inversion -- the
  typed-exception hint (TypeError+null -> "optional chaining")
  is more useful than the slot hint (mounted hook -> nextTick)
  for the same input.
- The Pyright `str | int` union type on `out[0]["text"]` caught
  by lint surfaces a real concern: the `dict[str, int | str]`
  type means we can't safely call string ops on the field
  without a cast. Wrapping in `str()` is the pragmatic fix;
  longer term a TypedDict per heading entry would be cleaner.
