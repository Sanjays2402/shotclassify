## Notable design decisions

- **tax_lines** intentionally excludes the bare `Tax` keyword because the top-level `tax` field owns the single-line case; `tax_lines` surfaces ONLY when 2+ jurisdictions appear so dashboards rely on `len(tax_lines) > 0` meaning "real multi-jurisdiction breakdown". Longest-first ordering combined with overlap-defence prevents `VAT` stealing the suffix of `Import VAT` (the matcher tracks `(start, end)` spans of already-recorded jurisdictions and skips overlapping matches).

- **gift_card_applied** always emits POSITIVE because the field semantic implies sign â€” the printer may write `Gift Card -25.00` (receipt convention: reductions are negative) or `Gift Card 25.00` (bare line) but both mean "25 was knocked off via gift card." Storing positive removes the sign ambiguity for reconciliation dashboards.

- **promo_code** rejects pure-digit codes longer than 3 digits because a 5-digit run after `Promo Code:` is almost always an order number that got misprinted under the wrong label. Short numeric codes (`Promo: 50`) accepted because some merchants legitimately use them. Original case preserved.

- **drivers_license** mirrors the passport mode's safety guarantee on every axis. The label-preserving substitution uses a custom `_sub_drivers_license` handler that replaces only the captured `num` group so the surrounding label stays visible â€” the reader knows the field WAS a license without the number leaking. 11+ digit runs that fail the regex tail are LEFT UNCHANGED as a safety property (better to skip than partially redact).

- **NestJS branch** sits BEFORE the generic Node branch because Nest runs on Node and the `at Foo.bar (file.ts:N:M)` frame shape is identical â€” without this discriminator a Nest exception would tag as `node` and dashboards would lose the framework-specific signal. The `[Nest]` PID prefix combined with the bracketed context tag is the unambiguous discriminator. When BOTH a typed exception class (e.g., `NotFoundException`) AND the bracketed context (e.g., `ExceptionsHandler`) are present, the typed class wins because dashboards care about the specific HTTP status code.

- **boto3 branch** runs FIRST inside the python branch of `parse_error_text`. When a Python traceback ALSO carries a boto signal (the typed exception header `botocore.exceptions.ClientError` OR the canonical `An error occurred (CODE)...` message), we return `framework='boto3'` immediately. Without this override the trace would tag as generic `python` and dashboards would lose the AWS-specific signal. Regular Python tracebacks (ValueError, KeyError, etc.) are untouched. The composed message slot reads `<detail> [code=ErrorCode op=OperationName]` so the structured AWS error code + operation pair survives the schema's plain-string `message` field â€” dashboards can pattern-match the `[code=...]` tail without re-running the regex.
